RAG知识库

RAG的作用和入门

RAG全称为 Retrieval Augmented Generation，翻译过来是检索增强生成。因为大模型训练结束之后不会再加入新的数据，但是用户肯定想要最新的数据，而且专业领域的知识大模型也是不清楚的，就需要外接知识库来增强大模型的生成能力。
大模型接入知识库后的工作流程是：

  1. 用户在应用中提出问题
  2. 在知识库寻找相关数据，返回相关的内容
  3. 组织prompt并发送给大模型
  4. 生成结果并返回响应
    知识库的存储数据一般使用的是向量数据库。简单来说就是知识库会把数据变成成百上千个维度，类似与向量的东西，然后计算向量的余弦相似度，将数据结构化就便于匹配数据。
    知识库还需要文本分割器把文档分成小的文本片段，这些片段又要借助向量模型，把它们转变成向量，然后存储到向量数据库中。

那么搜索知识库的过程就是，用户提交消息，向量模型吧消息转换成向量与数据库中比对，计算余弦相似值，越大说明越相关度越高，我们可以设置相似度高于多少才可被检索，检索出来的信息一起发给大模型，生成结果。

RAG使用到的核心api

1. 文档加载器

作用是把磁盘或者网络中的数据加载进程序，常见的有三类：

  • FileSystemDocumentLoader, 根据本地磁盘绝对路径加载
  • ClassPathDocumentLoader，相对于类路径加载
  • UrlDocumentLoader，根据url路径加载

2. 文档解析器

用于解析文档中的内容，把原本非纯文本数据转化成纯文本。比如初始的文档是pdf格式的，可以把它转为文本格式。
常见的有以下：

  • TextDocumentParser，解析纯文本格式的文件
  • ApachePdfBoxDocumentParser，解析pdf格式文件
  • ApachePoiDocumentParser，解析微软的office文件，例如DOC、PPT、XLS
  • ApacheTikaDocumentParser（默认），几乎可以解析所有格式的文件（在纯PDF文件方面的表现没有那么优秀）

使用方法：引入依赖，在EmbeddingStore中指定解析器

3. 文档分割器

常见的有以下

  • DocuemntByParagraphSplitter，按照段落分割文本
  • DocumentByLineSplitter，按照行分割文本
  • DocumentBySentenceSplitter，按照句子分割文本
  • DocumentByWordSplitter，按照词分割文本
  • DocumentByCharacterSplitter，按照固定数量的字符分割文本
  • DocumentByRegexSplitter，按照正则表达式分割文本
  • DocumentSplitters.recursive(…)(默认)，递归分割器,优先段落分割，再按照行分割，再按照句子分割，再按照词分割

4. 向量模型

前面用到的向量模型没有那么精准，可以自己外接专业的模型。
使用方法如下：

  • 在配置文件配置向量模型
  • 在EmbeddingStore和ContentRetriever指定向量模型（存储和检索用）

5. 向量数据库

EmbeddingStore虽然可用，但是数据都是存在内存里的，不太符合实际开发需求，所以要自己编写一个，将数据存到外部数据库中。
使用方法如下（使用redis数据库）:

  1. 部署一个扩展了redissearch的redis
  2. 引入依赖
  3. 在配置文件配置向量数据库连接信息
  4. 注入RedisEmbeddingStore对象使用

使用前的准备工作

  1. 准备数据库及数据表，并编写配置
  2. 引入数据库相关依赖（lombok，mysql，mybatis）
  3. 准备实体类
  4. 开发service层和mapper层，完成所需要的服务和对数据库的操作

使用tools让大模型帮我们调用函数

  1. 准备tools方法

Tool注解用于对方法的作用进行描述，P注解用于对方法的参数进行描述，使LangChain4j能通过反射的方式获取到Tool注解中的内容一并发送给大模型。
2. 配置tools方法
在@aiservice注解上加一行tools = “reservationTool”
